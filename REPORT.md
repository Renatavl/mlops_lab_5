# Звіт до ЛР5: Оркестрація ML-пайплайнів

## 1. Короткий опис реалізації

У `mlops_lab_5` реалізовано інтегровану MLOps-систему, що поєднує:

- DVC для керування стадіями підготовки/тренування,
- MLflow для трекінгу та реєстрації моделей,
- Apache Airflow для оркестрації DAG,
- Docker (multi-stage) для контейнеризації,
- GitHub Actions для CI перевірок.

### Реалізовані кроки з ТЗ

1. **Контейнеризація (Docker)**
   - Створено multi-stage `Dockerfile`.
   - Фінальний образ базується на `python:3.11-slim` і містить лише необхідні залежності.

2. **Розгортання оркестратора (Airflow)**
   - Створено `docker-compose.yaml` для `airflow-webserver`, `airflow-scheduler`, `postgres`, `airflow-init`.
   - Налаштовано монтування проєкту та DAG-ів у контейнери.

3. **Проєктування DAG**
   - DAG `ml_training_pipeline` виконує:
     - Sensor/Check даних,
     - Data preparation,
     - Model training,
     - Evaluation & Branching,
     - Model registration у MLflow зі стадією `Staging`.

4. **CI (GitHub Actions)**
   - Створено `.github/workflows/main.yaml`.
   - Реалізовано:
     - lint,
     - DAG integrity test,
     - Docker build test.

5. **Артефакти для Quality Gate**
   - Після тренування створюються:
     - `model.pkl`,
     - `metrics.json`,
     - `confusion_matrix.png`.

## 2. Архітектурні рішення

- Для задач DAG обрано `BashOperator` (prepare/train) як найпростіший і прозорий спосіб запуску існуючих скриптів.
- Логіку гілкування реалізовано через `BranchPythonOperator`, де рішення приймається за `accuracy_test` з `metrics.json`.
- Реєстрацію моделі винесено в окрему python-задачу DAG, що спрощує аудит і повторне використання.
- Тестування DAG імпорту реалізовано окремим CI-тестом (`DagBag`), щоб відловлювати синтаксичні/імпортні помилки до деплою.

## 3. Контрольні питання

1. **Як взаємодіють CI/CD (GitHub Actions) та Orchestrator (Airflow)? Що запускає що?**
   - CI перевіряє якість коду, коректність DAG і збірку образу при push/PR.
   - Airflow оркеструє runtime-виконання ML-конвеєра за розкладом/тригером.
   - Типовий потік: розробник змінює код → CI валідує → після успішного мерджу Airflow запускає оновлений DAG.

2. **Які переваги дає DAG проти лінійного скрипта?**
   - Явні залежності між задачами.
   - Керовані retry/timeout політики.
   - Спостережуваність статусів кожного кроку.
   - Гнучке розгалуження і відновлення з конкретного етапу.

3. **Навіщо `BranchPythonOperator` і де використовується в MLOps?**
   - Для реалізації policy-driven рішень у конвеєрі (Quality Gate).
   - Типові сценарії: реєстрація моделі лише при проходженні порогу метрик, вибір fallback-гілки, алертинг.

4. **Як multi-stage build впливає на безпеку та швидкість?**
   - Менший фінальний образ зменшує attack surface.
   - Відсутність зайвих build-інструментів у runtime-контейнері.
   - Краща кешованість шарів прискорює повторні збірки.

5. **Чому тестувати DAG у CI? Які помилки це ловить?**
   - Виявляє синтаксичні помилки,
   - проблеми імпортів/залежностей,
   - невалідні DAG-об'єкти,
   - помилки, що перешкоджають парсингу Airflow Scheduler.

6. **Ролі Scheduler, Webserver, Executor в Airflow**
   - `Scheduler`: парсить DAG-и, планує і ставить задачі в чергу.
   - `Webserver`: UI для моніторингу, тригерів, логів.
   - `Executor`: механізм фактичного запуску задач (локально/через Celery/Kubernetes).

7. **Що таке XComs і чому не передавати великі датасети через XCom?**
   - XCom — канал обміну малими службовими даними між тасками.
   - Великі об'єкти перевантажують metadata DB, сповільнюють scheduler/webserver і погіршують стабільність.
   - Для великих даних потрібно використовувати зовнішні сховища/артефакти (S3, файлові артефакти, DVC).

8. **Різниця між Sensor і звичайним Task (operator)?**
   - Sensor очікує на подію/умову (дані, файл, зовнішній сигнал).
   - Звичайний operator виконує обчислювальну/бізнес-операцію.
   - Приклад: `check_data_availability` сенсор очікує файл `weatherAUS.csv`.

9. **Що таке ідемпотентність задач DAG і чому це критично?**
   - Ідемпотентність означає, що повторний запуск задачі не змінює результат неконтрольовано.
   - Критично через retry, backfill, re-run після збоїв.
   - Без ідемпотентності виникають дублікати артефактів, неконсистентні метрики та помилки аудиту.
